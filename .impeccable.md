## Design Context

### Users

**Primary:** Rochel (the teacher) — creates and edits sheet music, manages song library, configures lessons using "The Batya Method."

**Secondary:** Young kids (ages 4-7) with a parent supervising, and older kids (7-12) who navigate independently. Also used in mixed-age teacher-led classroom/lesson settings.

The app serves dual roles: a **teaching tool** (Rochel prepares materials) and a **learning tool** (students interact with color-coded notation, playback, and guided tutorials). The editor must be powerful enough for the teacher while the learn mode and playback must be approachable for a child.

### Brand Personality

**Playful, warm, encouraging** — like a friendly piano teacher who celebrates every small win. The interface should feel safe to explore (no fear of breaking things) and rewarding to use (clear feedback, gentle celebrations on progress).

### Aesthetic Direction

- **Light mode only** — bright, clean, optimistic
- **Fredoka** as the primary typeface — rounded, soft, kid-friendly
- **Rainbow note colors** (C=Red through B=Pink) are a core pedagogical tool, not decoration
- **Purple** for educational/tutorial overlays (tours, learn mode)
- **Emerald green** (`#10b981`) as the primary accent — used for playback, success, and calls to action
- **Amber/gold** for interactive tutorial prompts
- Soft shadows, generous rounded corners (8-16px), subtle backdrop blurs
- **Reference feel:** Simply Piano / Yousician — gamified music learning with progress tracking, but simpler and less gamified. The current direction is good and needs polish, not reinvention.
- **Anti-reference:** Avoid overly corporate/minimal music apps (Flat.io, Noteflight). Avoid dark themes. Avoid dense professional notation software aesthetics (Sibelius, Finale).

### Design Principles

1. **Clarity over cleverness** — Every UI element should be immediately understandable to a child. Prefer explicit labels and icons over abstract symbols. When in doubt, make it obvious.

2. **Color is pedagogy** — The rainbow note colors are not styling choices; they're teaching tools. Never override, mute, or compete with note colors. Supporting UI should be neutral to let note colors stand out.

3. **Safe to explore** — Undo is always available, confirmations protect destructive actions, and errors are gentle ("Oops! Let's try again" not "Error: invalid input"). Kids and teachers should feel confident clicking around.

4. **Celebrate progress** — Use motion and visual feedback to mark achievements. Stage completions, correct note placements, and lesson milestones deserve small moments of delight (confetti, green checkmarks, encouraging copy).

5. **Teacher-ready, kid-friendly** — The editor needs professional capabilities (export, settings, time signatures) without overwhelming young users. Use progressive disclosure: simple by default, powerful when needed.

### Typography

- **Primary font:** Fredoka (weights 300-700) — all UI text
- **Music notation:** Leland Text (SMuFL) — noteheads, rests, clefs, accidentals
- **Backup music font:** Bravura (SMuFL)

### Accessibility (Current Stance)

- Focus on usability for sighted children for now
- Color-coded notes rely on color as the primary differentiator (shapes/patterns deferred)
- Standard keyboard navigation where applicable
- No specific WCAG target yet — keep contrast reasonable
